# Standard Library — Design Spec

**Track 4** (see `docs/ROADMAP.md`). Depends on collections (done),
exceptions (done), the module system (done).

## Goal

A Python-style "batteries included" standard library: a set of `.st` modules
loadable through the module system, covering what every program needs.
Each module ships with tests. The library is community-extensible — one module
is one self-contained contribution.

## What exists today

The module system (`Import from: '…'`, F5/F5v2) loads `.st` files and routes
through protoCore's UMD. But there is **no standard library** — no `lib/`
directory, and the runtime has no default place to find stdlib modules.
Everything today is a C++ primitive or user code.

## Architecture

### The `lib/` directory + discovery

Standard-library modules live in `lib/` at the project root, as `.st` files
(`lib/stream.st`, `lib/json.st`, …). The runtime must find them without the
user setting `STPATH`:

- The `STRuntime` adds a standard-library directory to its default module
  resolution chain (after F5v2's `provider:st`, before/around the cwd).
- Discovery: an env var `PROTOST_LIB` if set; otherwise a path derived from
  the executable location (a `lib/` sibling of the `protost` binary, or the
  build/source `lib/`). Pick a scheme that works for both a dev build and an
  installed binary; document it. The venv may also carry/point at a `lib/`.
- `Import from: 'stream'` then resolves `lib/stream.st` with no STPATH.

### Module shape

A stdlib module is an `.st` file that defines classes/methods, idiomatic
protoST. Two kinds:

- **Pure modules** — written entirely in protoST over the existing language
  and collections (e.g. `Stream`, `JSON`). No new primitives.
- **Primitive-backed modules** — need the OS or libm (clocks, `sqrt`, OS
  randomness, file I/O). The low-level operations are C++ primitives
  bootstrapped into the runtime (always available); the `.st` module wraps
  them in an idiomatic API. There is no per-module primitive loading — the
  primitives are bootstrapped, the `.st` layer is the loadable idiomatic face.

### Float arithmetic prerequisite

`docs/LANGUAGE.md` §"Known deviations" D11 records that arithmetic is bound on
`SmallInteger` only — `Float` / `LargeInteger` / mixed-mode arithmetic is
missing. `Math` (and any numeric module) needs working `Float`. Closing D11 —
binding `+ - * / < > = …` on `Float` and handling Int/Float mixed-mode — is
part of this track (the `Math` sub-slice).

## Sub-slices

Land in order; each is a module (or the infrastructure) with its own tests.

- **T4-a — stdlib infrastructure + `Stream`.** Establish `lib/` and default
  discovery. First module: `Stream` — `ReadStream` / `WriteStream` over a
  collection (`next`, `next:`, `peek`, `atEnd`, `upToEnd`, `contents`,
  `nextPut:`, `nextPutAll:`, `<<`). Pure protoST over Track 2 collections —
  it validates the infrastructure end to end.
- **T4-b — Math protocol + Float arithmetic.** Close D11: bind arithmetic and
  comparison on `Number`, with Int/Float mixed-mode coercion (done by MNT-c).
  Then the math protocol: `sqrt`, `sin`/`cos`/`tan` (+ inverses),
  `ln`/`exp`/`log`/`log:`, `floor`/`ceiling`/`rounded`/`truncated`,
  `sign`/`squared`/`reciprocal`/`isZero`/`min:`/`max:`/`between:and:`/
  `asFloat`/`asInteger`/`even`/`odd`/`gcd:`/`lcm:`, exact `raisedTo:` and
  `factorial`, and class-side `Float` constants `pi`/`e`/`infinity`/`nan`.

  **Decision: Math is bootstrap, not a loadable `lib/math.st` module.** The
  idiomatic Smalltalk form is a unary / keyword message on a number — `9 sqrt`,
  `x sin`, `30 factorial`, `2 raisedTo: 100` — which must work with no
  `Import`. So the math operations are C++ primitives bootstrapped onto the
  shared `Number` prototype (the transcendentals thin wrappers over libm), and
  the constants are bound class-side on `Float`. There is no `lib/math.st`;
  the math messages are always available, exactly like `+` and `printString`.
  A libm domain error (`(-1) sqrt`, `0 ln`) is let through as its IEEE-754
  result (`nan`/`inf`) — the protocol is total and never raises a domain
  error; only invalid arguments (negative `factorial`, `gcd: 0 0`) raise.
- **T4-c — `Random`.** *(landed)* A seedable pseudo-random generator —
  `lib/random.st`, a pure-protoST PRNG so it is deterministic and needs no
  primitive: `Random seed: n`, `Random new` (fixed default seed), `next` (a
  Float in [0,1)), `nextInt:` (1..n), `between:and:`, `next:` (an Array of n
  draws). Implemented as a 32-bit LCG (Numerical Recipes constants); the LCG
  was chosen over xorshift because it needs only integer `* + \\` and no
  bitwise operators, and its intermediate product stays inside the 56-bit
  SmallInteger range. `atRandom` on `Interval`/collections was skipped — it
  would require touching the Track 2 collection classes, which is out of
  scope for this slice.
- **T4-d — `JSON`.** *(landed)* Parse and generate JSON, pure protoST:
  `JSON parse:` (a string → protoST objects: `Dictionary`, `Array`, strings,
  numbers, booleans, nil), `JSON stringify:` (the inverse). `lib/json.st` — a
  hand-written recursive-descent scanner; numbers are accumulated digit by
  digit (an integral mantissa with no fraction/exponent yields an integer that
  promotes to `LargeInteger` on overflow, otherwise a `Float`); JSON string
  escapes (`\" \\ \/ \n \t \r \b \f \uXXXX`) are honoured on parse and emitted
  on stringify. Invalid input signals a catchable `Error` naming the position.

  The String protocol turned out to expose **no character access at all** —
  only `, size = ~= printNl` — so the slice's own implementation notes (a
  `ReadStream` over the input, `aChar` → its value) were not directly
  satisfiable. Two minimal, idiomatic enabling changes were made rather than a
  hack: (1) `String>>at:` (n-th character as a 1-char String) and
  `String>>asInteger` (codepoint of the first character), plus the inverse
  `Number>>asCharacter`, all UTF-8-codepoint-aware C++ primitives — and
  `String>>size` was made codepoint-based to match `at:`; (2) `String` and
  `Boolean` were registered as globals (their prototypes already existed) so
  the module can extend them with `jsonStringOn:` double-dispatch methods,
  the language-idiomatic substitute for the absent class-test predicates
  (`isKindOf:` / `class`). `stringify:` dispatches by double dispatch.
- **T4-e — `Time`** *(landed)*. Wall-clock access. Two C++ clock primitives
  bootstrapped onto `Object` — `__currentMillis` (epoch milliseconds from
  `std::chrono::system_clock`, the wall clock) and `__monotonicMillis`
  (milliseconds from `std::chrono::steady_clock`, for elapsed-time
  measurement) — in `src/primitives/time_prims.cpp`. On top of them,
  `lib/time.st` provides an idiomatic object model: `Time class >> now` →
  a `Timestamp`; `Time class >> millisecondsToRun:` → elapsed monotonic
  milliseconds for a block. `Timestamp` wraps epoch milliseconds
  (`asMilliseconds` / `asSeconds`, comparison, `Timestamp - Timestamp` → a
  `Duration`, `Timestamp + Duration` → a `Timestamp`); `Duration` wraps a
  millisecond span (class-side `seconds:` / `milliseconds:` / `minutes:`,
  arithmetic and comparison between durations). Deliberately minimal — no
  timezones, no calendar; a `Timestamp` is an opaque instant and a
  `Duration` an opaque span. Pure protoST over the two primitives.

## Tests

Each module ships with conformance-style and unit tests:
- A module's behaviour gets `.st` conformance tests under
  `tests/conformance/` (a new `13-stdlib/` section) — and the module is
  exercised via `Import from: '…'`.
- Per-module unit tests where finer assertions help.
- The discovery mechanism is tested: `Import from: 'stream'` works with no
  `STPATH` set.
- Regression: the existing 557-test suite stays green.

## Out of scope

File I/O and networking modules — they touch larger OS-surface and security
questions; they belong to a later batch or the digital-twin layer. This track
delivers the in-process "batteries": streams, math, random, JSON (and maybe
time).
