## Actor-model benchmarks

protoST-specific — protoPython has no actor model. These exercise the cooperative actor scheduler.

| Benchmark | Result |
|---|---|
| **Parallel speedup** | 12 CPU-bound worker actors: 307 ms with the full pool vs 581 ms with `PROTOST_WORKERS=1` — **1.89× speedup** |
| **Cooperative-yield scaling** | **1000** waiter actors, each parked on a nested `wait`, all hosted on **K=2** worker threads — completes in 970 ms. Thread-per-actor blocking would need 1000 OS threads. |
| **Message throughput** | 2,000 drained round-trip sends to one actor in 31 ms — **63,794 messages/second**. |

### Reading these numbers

protoST single-thread sits at **geomean 2.60× CPython 3.14
free-threading** and **0.54× protopy** (i.e. ~1.85× faster than its
companion Python runtime) on this 7-test comparable suite. There is
one outright win vs CPython (`str_concat` 0.62×), one parity
(`int_sum_loop` 1.09×), and 4 wins vs protopy
(`str_concat` 0.05×, `list_append` 0.25×, `range_iterate` 0.29×,
`attr_lookup` 0.45×). The actor results remain the strategic
differentiator: 1000 suspended actors on 2 worker threads
(thread-per-actor cannot do), 1.89× speed-up on the parallel
benchmark from extra cores with zero code change, 63.8 K msg/s
mailbox throughput on a notebook.

The 30.89× on `exception_latency` is the remaining headline gap on
single-thread. It comes from a `[…] on: Error do: [:e | …]` pattern
fired 50 K times — each iteration's signal goes through a C++
`throw UnwindToHandler` that costs ~6 % of total CPU in the unwind
infrastructure (`__gxx_personality_v0` + `_Unwind_Find_FDE`). Closing
that gap requires inlining the protected/handler blocks into the
calling engine so the throw is replaced by a bytecode jump — a
multi-week refactor (handler-PC tables, runLoop catch-and-resume,
signal-throwing code rework). Scoped out of this sprint; the
single-guard fast path landed today gives ~3 % on the workload,
leaving the structural cost intact.

### What's left for the next sprint

Ranked by ROI / risk:

1. **`on:do:` inline (multi-week)** — would attack the remaining
   `exception_latency` 30.9× CPython. Requires handler-PC tables
   in `BytecodeModule`, runLoop catch-and-resume on
   `UnwindToHandler`, signal-throwing code that finds handlers by
   frame-stack walk instead of C++ throw. The right way to do it
   is design-first.

2. **SEND inline cache for fib-shape recursion** — `fib` stays
   7.6× CPython; the remaining hot cost is `getAttribute` MRO walks
   for `compute:` resolution. An IC keyed on receiver class would
   short-circuit the walk. Investigated and scoped out this sprint
   because protoCore does not expose a stable per-class
   discriminator API cleanly — the cleanest path is a small
   protoCore addition (`ProtoObject::getClassIdentity()` returning
   the prototype cell pointer) and then a per-bytecode-position IC
   slot in `BytecodeModule`.

3. **Per-actor message slab allocator** — would attack the
   remaining BEAM messaging gap. ~5-10× possible on
   multi-producer patterns. Architecturally significant; not on
   any user's critical path.
