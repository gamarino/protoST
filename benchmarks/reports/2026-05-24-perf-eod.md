## Actor-model benchmarks

protoST-specific — protoPython has no actor model. These exercise the cooperative actor scheduler.

| Benchmark | Result |
|---|---|
| **Parallel speedup** | 12 CPU-bound worker actors: 307 ms with the full pool vs 581 ms with `PROTOST_WORKERS=1` — **1.89× speedup** |
| **Cooperative-yield scaling** | **1000** waiter actors, each parked on a nested `wait`, all hosted on **K=2** worker threads — completes in 970 ms. Thread-per-actor blocking would need 1000 OS threads. |
| **Message throughput** | 2,000 drained round-trip sends to one actor in 31 ms — **63,794 messages/second**. |

### Reading these numbers

protoST single-thread sits at **geomean 2.60× CPython 3.14
free-threading** and **0.54× protopy** (i.e. ~1.85× faster than its
companion Python runtime) on this 7-test comparable suite. There is
one outright win vs CPython (`str_concat` 0.62×), one parity
(`int_sum_loop` 1.09×), and 4 wins vs protopy
(`str_concat` 0.05×, `list_append` 0.25×, `range_iterate` 0.29×,
`attr_lookup` 0.45×). The actor results remain the strategic
differentiator: 1000 suspended actors on 2 worker threads
(thread-per-actor cannot do), 1.89× speed-up on the parallel
benchmark from extra cores with zero code change, 63.8 K msg/s
mailbox throughput on a notebook.

The 30.89× on `exception_latency` is the remaining headline gap on
single-thread. It comes from a `[…] on: Error do: [:e | …]` pattern
fired 50 K times — each iteration's signal goes through a C++
`throw UnwindToHandler` that costs ~6 % of total CPU in the unwind
infrastructure (`__gxx_personality_v0` + `_Unwind_Find_FDE`). Closing
that gap requires inlining the protected/handler blocks into the
calling engine so the throw is replaced by a bytecode jump — a
multi-week refactor (handler-PC tables, runLoop catch-and-resume,
signal-throwing code rework). Scoped out of this sprint; the
single-guard fast path landed today gives ~3 % on the workload,
leaving the structural cost intact.

### What's left for the next sprint

### What perf work IS legitimate to keep doing

Today's wins came entirely from changes that do not touch the
conceptual model: compiler inlining of common conditional/loop
patterns, dead-code elimination of debug-only `getenv` and `toImpl`
checks, lazy caching of statically-derivable values, idiomatic use
of protoCore facilities (symbol table for canonical Dictionary
key hashing, SmallInt tag helpers for fast-path arithmetic). None
of those introduce special-cased subsystems; each makes existing
paths cheaper without changing what an existing path does.

The same shape of work has more room:

1. **`on:do:` inline of the protected and handler blocks** into
   the calling engine (handler-PC tables in `BytecodeModule`,
   runLoop catch-and-resume on `UnwindToHandler`). Would attack
   `exception_latency` 30.9× CPython by replacing the per-iter
   C++ `throw UnwindToHandler` with a bytecode jump. Multi-week
   refactor of the exception path; design-first is mandatory.

2. **SEND inline cache for monomorphic call sites** (fib-shape
   recursion). Cache `{receiver class, resolved method}` per SEND
   bytecode position. Investigated and scoped out this sprint
   because the natural per-class discriminator is the prototype
   cell pointer and protoCore's public API does not expose it
   cleanly — a small protoCore addition
   (`ProtoObject::getClassIdentity()`) opens the door.

Both above keep the conceptual model uniform and reduce cost on
existing paths. They are real optimisation candidates whenever
the perf budget warrants them.

### What perf work to NOT do (yet)

Earlier drafts of this section listed a third item — a per-actor
message slab allocator, ring buffer mailbox, lightweight
non-Cell future struct — that would close the residual gap to
BEAM on shallow-message microbenchmarks. After the discussion at
end-of-sprint, that item is **deliberately not pursued** for the
following reasons:

* **No real user is asking for it.** protoST is pre-production.
  Closing a hypothetical micro-benchmark gap against another
  runtime's headline number is optimisation against the wrong
  metric.

* **The headline metric itself is misleading.** BEAM's 5–10 M
  msg/s figure comes from shallow-payload ping-pong benchmarks
  on per-process heaps. In workloads protoST is shaped for —
  digital twins, agent simulation, anything that ships a 10K-node
  world graph between actors — BEAM deep-copies the message
  graph between process heaps, which costs milliseconds per send
  and serialises against any "shared world" pattern. protoST
  passes a pointer to a structurally-shared immutable snapshot;
  the same send is ~17 µs and a hundred receivers reading the
  same snapshot pay zero contention. The "honest" comparator on
  protoST's actual value proposition is favourable, not adverse.

* **The proposed redesign would damage the conceptual model.**
  A ring-buffer mailbox breaks the uniform "mailbox is a
  `ProtoList` you can `size` / `first` / snapshot" property. A
  fixed-struct envelope breaks "every message is a regular
  object". A lightweight future struct breaks "every Future
  composes like a regular ProtoObject". Each is a
  special-cased actor subsystem that pulls the model toward Akka
  / BEAM territory and away from the Smalltalk ideal — "one way
  to think about objects, including messages, including mailboxes,
  including futures". The cost is permanent and architectural;
  the gain is contingent on benchmarks that do not reflect the
  target workload.

* **Even the perf math does not survive scrutiny.** A re-run with
  honest accounting (per-thread arenas already give bump-pointer
  allocation, so the "slab" is redundant on the allocator side;
  the receiver bytecode floor is ~5-10 µs per message regardless)
  put the realistic ceiling at 1.5-2× the current numbers, not
  the 5-10× initially projected — and that gain is contingent on
  ALL the structural changes landing together.

**Defer.** When a real user surfaces a workload where actor
messaging is the measured bottleneck — and where the structural
redesign would survive their requirements — revisit with their
data. Until then, treat current actor messaging perf (60-70 K
msg/s shallow, 3.88× scaling on 6 cores, 1000-actor cooperative
density) as **structurally adequate for the design-target
workloads** and continue investing in correctness, tooling,
ergonomics, and finding that first concrete user.
