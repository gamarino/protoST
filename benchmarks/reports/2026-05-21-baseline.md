## Actor-model benchmarks

protoST-specific — protoPython has no actor model. These exercise the cooperative actor scheduler.

| Benchmark | Result |
|---|---|
| **Parallel speedup** | 12 CPU-bound worker actors: 6511 ms with the full pool vs 12816 ms with `PROTOST_WORKERS=1` — **1.97× speedup** |
| **Cooperative-yield scaling** | **1000** waiter actors, each parked on a nested `wait`, all hosted on **K=2** worker threads — completes in 8458 ms. Thread-per-actor blocking would need 1000 OS threads. |
| **Message throughput** | _not run — opt-in (`--with-throughput`); see the known issue below._ |

### Known issue — actor-scheduler deadlock under sustained mailbox load

The message-throughput benchmark (`benchmarks/actors/message_throughput.st`) is **kept out of the default timed path** and only runs with `--with-throughput`. Sending many messages to one actor — fire-and-forget *or* drained — hangs the runtime **non-deterministically**: repeated runs alternate between sub-second completion and indefinite stalls, and a *larger* message count sometimes completes where a smaller one hung. This is a scheduling / GC race under sustained mailbox pressure, not a clean N threshold. It is a real runtime bug — tracked as **D23** in `docs/STATUS.md` — and the benchmark file is retained as its repro. The two actor benchmarks above (parallel speedup, cooperative-yield) are stable and are the reported actor results.

### Reading these numbers

protoST's single-thread arithmetic is slower than CPython — it is a young runtime and the comparable table shows that honestly. The actor results are the differentiator: the cooperative-yield benchmark hosts a thousand suspended actors on two OS threads, which a thread-per-actor model fundamentally cannot do, and the parallel benchmark turns extra cores into real wall-clock speedup with no code change.
