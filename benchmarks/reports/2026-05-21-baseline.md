## Actor-model benchmarks

protoST-specific — protoPython has no actor model. These exercise the cooperative actor scheduler.

| Benchmark | Result |
|---|---|
| **Parallel speedup** | 12 CPU-bound worker actors: 6511 ms with the full pool vs 12816 ms with `PROTOST_WORKERS=1` — **1.97× speedup** |
| **Cooperative-yield scaling** | **1000** waiter actors, each parked on a nested `wait`, all hosted on **K=2** worker threads — completes in 8458 ms. Thread-per-actor blocking would need 1000 OS threads. |
| **Message throughput** | **2000** drained round-trip sends to one actor in ~0.26 s — **~7,700 messages/second** (enqueue → schedule → execute → settle Future → wake sender). |

### Resolved — actor-scheduler deadlock under sustained mailbox load (D23)

> **Update (post-baseline):** the deadlock described here is **fixed** —
> `docs/STATUS.md` D23, Closed. The original text is kept for the record.

The message-throughput benchmark (`benchmarks/actors/message_throughput.st`) was originally **kept out of the default timed path** because sending many messages to one actor — fire-and-forget *or* drained — hung the runtime **non-deterministically**: repeated runs alternated between sub-second completion and indefinite stalls. Root cause: `STRuntime::workerLoop` acquired the scheduler mutex `schedMu` with a plain (non-GC-safe) lock, so a worker could block off-safepoint as a counted mutator and stall the stop-the-world GC quorum — a worker ↔ schedMu-holder ↔ GC deadlock. The fix routes `workerLoop`'s `schedMu` acquisition through `gcSafeLock`, like every other acquirer. Verified across 60+ repeated repro runs and a `cli_actor_stress` regression test; the throughput benchmark now runs and is reported in the table above. The two actor benchmarks (parallel speedup, cooperative-yield) were unaffected and remain stable.

### Reading these numbers

protoST's single-thread arithmetic is slower than CPython — it is a young runtime and the comparable table shows that honestly. The actor results are the differentiator: the cooperative-yield benchmark hosts a thousand suspended actors on two OS threads, which a thread-per-actor model fundamentally cannot do, and the parallel benchmark turns extra cores into real wall-clock speedup with no code change.
